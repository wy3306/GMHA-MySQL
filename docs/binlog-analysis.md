# Binlog 分析

Binlog 分析位于 **集群管理 → 实例管理 → binlog分析**。功能参考
`wy3306/bin2sql` 的事务分析能力，通过 MySQL 复制协议读取已登记实例的
Binlog，生成 DML/DDL 统计、写入趋势、热点表和大事务报告，并利用 MySQL 8
GTID 提交时间还原历史复制延迟。

## 使用方法

1. 选择集群中的 MySQL 实例。
2. 设置开始与结束时间。建议生产环境先从近 30 分钟或近 1 小时开始，单次
   最长 7 天。
3. 起始 Binlog 文件通常留空，由系统按时间自动定位。云数据库不允许
   `SHOW BINARY LOGS` 时，可以填写文件名绕过自动定位。
4. 选择大事务判定方式：
   - 按影响行数：适合批量 INSERT、UPDATE、DELETE。
   - 按事务字节：使用 GTID 事件中的 `transaction_length`，源端没有该字段时
     可能无法识别。
5. 创建任务后查看实时文件进度；完成后可切换热点表、大事务、DDL 和 DML
   明细。大事务页会在元数据可用时显示历史复制延迟，全部结果可导出 JSON。

## 实现与安全

- Manager 使用已启用的 MHA 管理账号连接实例；API 不接收用户名或密码，
  页面和任务结果也不会返回凭据。
- 分析只读取复制流，不执行数据或结构变更。
- 多个 Binlog 文件会并行处理，并根据 Manager CPU 负载、内存占用和文件数
  自动降低并发。
- DML 明细最多保留 20,000 条，DDL 明细最多保留 5,000 条，大事务最多保留
  5,000 条；达到上限时只截断明细，汇总统计仍然保留。
- Manager 同时最多运行两个 Binlog 分析任务，并保留最近 50 条任务记录。

## 注意事项

- 账号至少需要读取 Binlog 列表和复制流的权限；GMHA 默认 MHA 账号已包含
  `REPLICATION CLIENT` 与 `REPLICATION SLAVE`。
- 分析会占用实例网络、Manager CPU 与内存。生产环境应避开高峰，并优先缩小
  时间窗口。
- DML 行数依赖 ROW 格式事件；非 ROW Binlog 无法提供同等精度。
- 按字节识别依赖 MySQL 版本和 GTID 元数据。结果为 0 时应改用行数阈值。
- 历史复制延迟依赖 GTID 事件中的原始和当前提交时间；旧版本或非 GTID
  日志不含该元数据时显示为不可用。
- 当前任务记录保存在 Manager 内存中，Manager 重启后不会保留历史结果；如需
  留档，请在完成后导出 JSON。
