# 数据库归档

数据库归档位于 **集群管理 → 实例管理 → 数据归档**。功能通过目标机器的
Agent 调用 Percona Toolkit 的 `pt-archiver`，使用 Agent 已登记的 MHA 管理
账号；浏览器请求、Manager 数据库和任务命令均不保存数据库密码。

## 执行流程

1. 选择实例，填写源表、归档表、归档条件和扫描索引。
2. 运行归档预检。系统检查 `pt-archiver`、源表、条件匹配行数、查询计划、
   表空间、主键和归档目标；目标表不存在时，预检会明确提示正式执行将自动创建。
3. 若预检通过，确认批次大小、限速、单次运行时长和是否删除源数据。
4. 勾选风险确认，并输入完整的
   `源库.源表->归档库.归档表` 后执行。
5. 正式任务依次准备目标表、执行 PT dry-run、分批归档，并核验源表剩余匹配
   行数及归档表总行数。

## 安全门禁

- 归档条件必填，且拒绝 `1=1`、多语句、SQL 注释、数据变更语句及
  `SLEEP`/`BENCHMARK` 等高风险函数。
- 源表和目标表不能相同。库名、表名和索引名仅接受安全的 MySQL 标识符。
- 正式执行必须建立在同一组参数的成功预检之上；任何目标、条件、批次、限速
  或归档模式变化都会使原预检失效。
- 预检匹配行扫描上限为 100,000 行；达到上限时页面显示 `≥ 100,000`，避免
  在大表上先执行无界 `COUNT(*)`。完整处理行数以 PT 执行统计为准。
- 默认每批 1000 行、批次间隔 1 秒、每批提交，并对锁等待/死锁重试 3 次。
- “归档后删除源数据”关闭时，命令使用 `--no-delete`，仅复制数据；开启时，
  PT 在目标表插入成功后删除对应源行。
- 目标表不存在时使用 `CREATE TABLE IF NOT EXISTS ... LIKE ...` 创建。目标库
  必须预先存在。

## 运维建议

- 条件应从扫描索引的低值端开始，例如时间列或主键范围；优先使用主键或能够
  支撑条件的唯一索引。
- 先在测试实例验证查询计划、目标表结构、字符集和重复键行为，再进入生产。
- 根据写入负载调低批次大小或增加批次间隔，并用“单次最长执行”控制维护窗口。
- 归档到同一实例会同时产生读、写、binlog 和额外磁盘占用；确认磁盘容量及
  复制延迟后再执行。
- PT 的源、目标连接使用不同数据库连接。事务型表也应通过预检和测试验证异常
  中断后的重跑行为。

参数语义和限制以
[Percona pt-archiver 官方文档](https://docs.percona.com/percona-toolkit/pt-archiver.html)
为准。
