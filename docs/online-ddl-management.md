# 在线 DDL 管理

在线 DDL 位于 **集群管理 → 实例管理 → 在线 DDL**。该功能使用目标机器上的
`pt-online-schema-change` 创建影子表、分块复制数据，并在最后以短时元数据锁完成
原表切换。所有操作均通过 Agent 的审计任务执行。

## 使用流程

1. 选择目标实例，填写数据库、数据表和 `ALTER TABLE` 后面的变更子句。
2. 填写变更目的和预期影响，按业务负载调整 PT 门禁参数。
3. 执行“PT 预演”。预演会检查 PT 工具、目标表、表引擎、唯一键、触发器、外键、
   当前负载与复制条件，并运行 `--dry-run`，不会修改原表。
4. 预演成功后，确认磁盘、复制压力、长事务和元数据锁风险，输入完整的
   `数据库.数据表`。
5. 执行在线 DDL。任务依次重新预检、重新 dry-run、执行 `--execute`，最后读取
   变更后的表结构进行核验。

只要目标实例、DDL 子句、用途说明或任一负载门禁发生变化，页面就会使旧预演结果
失效，必须重新预演。

## 默认门禁

| 参数 | 默认值 | 行为 |
|---|---:|---|
| `Threads_running` 暂停阈值 | 25 | 达到阈值时暂停分块复制 |
| `Threads_running` 中止阈值 | 50 | 达到阈值时终止任务 |
| 最大复制延迟 | 10 秒 | 超过阈值时等待副本追平 |
| 目标块耗时 | 0.5 秒 | PT 自动调整每批复制行数 |
| 检查间隔 | 1 秒 | 暂停期间的重新检查间隔 |
| 外键处理 | `auto` | 由 PT 根据表规模选择策略 |

## 安全边界

- API 只接受单表 ALTER 子句，不接受完整 SQL、语句分隔符、SQL 注释、显式
  `LOCK`/`ALGORITHM` 或表重命名。
- 正式执行必须通过服务端风险确认和精确目标确认，不能只依赖页面校验。
- MySQL 账号和密码不由浏览器提交。Agent 从已登记的实例配置创建权限为 `0600`
  的临时 defaults file，任务完成后立即删除。
- 正式执行始终再次运行 dry-run，避免预演后环境变化直接进入数据复制。
- PT 复制进度会持续写入任务状态；完整输出、失败原因与步骤结果可在任务中心查看。

## 运行前准备

目标机器必须已安装 Percona Toolkit，并能从 `PATH` 找到
`pt-online-schema-change`。创建 MySQL 实例时可勾选“安装 PT 工具”，也可通过
Manager 的 `percona-toolkit` 离线制品流程部署。

生产执行前应确认：

- 可用磁盘足以容纳影子表和相关索引；
- 表具有主键或合适的唯一索引；
- 已评估已有触发器、外键和复制过滤；
- 没有会长期持有元数据锁的事务；
- 业务和复制链路可承受复制期间的额外读写与 binlog 压力；
- 已准备备份、监控和人工中止后的处理方案。
